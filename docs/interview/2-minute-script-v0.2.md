# 2-Minute Interview Script (v0.2)

## Goal of this version
This version is more conversational and easier to deliver naturally while still sounding technical and credible.

## Delivery structure (simple)
1. Problem
2. What I built
3. Why it is reliable
4. Why it matters for the role

## Core script (natural delivery)
I’m currently building a project called the Public Renewable Grid Resilience Auditor.

The problem I’m solving is that public-sector energy planning data is usually fragmented, which makes it hard to track renewable progress and reliability risk in a consistent way.

So I designed an end-to-end analytics platform using free data sources like ENTSO-E, Open-Meteo, and Ember. Technically, I’m using a Bronze/Silver/Gold architecture on DuckDB and Parquet, with contract-first ingestion, quality rules, and reproducible validation workflows.

Up to now, I’ve completed governance and KPI definitions, environment and CI standards, source contracts, and raw ingestion jobs with retry/backoff and idempotent metadata-aware writes.

What I think is strongest about this project is that it combines technical execution with decision usefulness: the outputs are meant for policy and planning, not just for engineering completeness.

I’m now moving into staging models, harmonization, and marts, and then forecasting, dashboard narrative, and orchestration.

## Opening hooks (pick one)

### Hook A (business-first)
I’m building a system that helps planners answer a simple question reliably: are we integrating renewables without increasing grid stress?

### Hook B (technical-first)
I’m building a contract-first analytics platform where data reliability is treated as a product feature, not an afterthought.

### Hook C (role-first)
I wanted a project that proves I can own the full analytics lifecycle from data contracts to stakeholder-ready insight.

## Delivery tips

- Do not list every tool unless asked.
- Emphasize outcomes and choices, not only tasks.
